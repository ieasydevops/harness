# Skill: 提取-Harness模板

触发：人工指令。从当前项目的真实 Harness 工程中，蒸馏出项目无关的通用模板，供新项目复用。

自动执行约束：在用户指定的目标目录下，新建文件/文件夹、更改文件内容时，AI 可自动执行，无需逐一向用户确认（如用命令行批量写入所有模板文件）。

## 上下文管理

源工程文件数量多（30+），禁止以下做法：
- 禁止通过子代理（use_subagents）读取源文件，子代理结果容易被截断导致内容丢失
- 禁止一次性读取全部源文件后再统一处理，避免上下文溢出

必须采用分批流水线：按目录分组，每组执行"读取 -> 蒸馏 -> 写入"，写入完成后再处理下一组。

## 步骤

### Step 1 -- 确认输出目录

向用户询问模板保存的目标目录（绝对路径）。

### Step 2 -- 分批读取、蒸馏、写入

主代理使用 read_file 逐个读取源文件完整内容，按以下分组顺序处理。每组完成"读取 -> 蒸馏 -> 写入目标目录"后，再处理下一组：

第 1 组：AGENTS.md、.harness/README.md
第 2 组：.harness/docs/ 下全部文件
第 3 组：.harness/skills/ 下全部文件
第 4 组：.harness/subagents/ 下全部文件
第 5 组：.harness/context/agents/ 下全部文件（含 .gitignore）
第 6 组：.harness/context/users/ 下全部文件

蒸馏规则（每个文件均适用）：

1. 剥离项目专属信息（项目名称、业务逻辑、技术栈细节、人员信息、绝对路径、UUID 等），替换为 `{{占位符名称}}` 格式
2. 保留通用框架、结构、格式、流程定义
3. 通用规范（Skills 机制、文件管理规则、知识库管理原则等）原文保留，不抽象
4. 项目规范（代码生成、架构边界、质量守护、安全规范等）仅保留结构骨架和通用条目，项目专属条目替换为占位符示例
5. Skills 文件：流程通用的原文保留，引用项目专属实体（文件名、类名、API 名）的部分替换为占位符
6. Subagents 文件：合并为一个示例模板（scan-example.md），用占位符标注扫描维度、范围、规则
7. Context 知识库文件：保留文档结构和章节标题，正文内容替换为占位符和填写提示
8. Context users 文件：保留文档结构，正文替换为占位符
9. README 文件：保留文档结构，正文替换为占位符，更新文件清单描述

文风约束：
- 禁用 emoji、加粗、斜体等润色格式
- 仅保留纯文本和 Markdown 基础结构（标题层级、列表、表格、代码块）

写入目标目录结构与源工程一致：

```
目标目录/
  AGENTS.md
  .harness/
    README.md
    docs/
    skills/
    subagents/
    context/
      agents/
      users/
```

### Step 3 -- 清理多余文件

1. 递归列出目标目录下的全部文件
2. 与本次生成的模板文件清单比对，找出不属于模板的多余文件
3. 排除 GitHub 项目管理文件（如 `.git/`、`.gitignore`、`LICENSE`、`.github/` 等），这些文件不视为多余
4. 如存在多余文件，列出清单并向用户确认是否删除
5. 用户确认后执行删除；用户拒绝的文件保留不动

### Step 4 -- 验证

1. 列出目标目录全部文件，确认无遗漏、无多余空文件
2. 在全部生成的 .md 文件中搜索项目专属关键词（项目名称、核心类名、API 名称、绝对路径等），确认零匹配
3. 如发现泄露，修正后重新验证

### Step 5 -- 输出报告

向用户输出：
- 生成的文件清单及数量
- 验证结果（项目专属关键词扫描结果）
- 占位符汇总（列出模板中使用的全部 `{{占位符}}` 名称）
